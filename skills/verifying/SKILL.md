# Verifying: приёмка возвращает факт, не мнение

Приёмка — отдельным верификатором (qa, чистый контекст): исполняет команду/сценарий из DoD, возвращает факт.

## Принцип

Верификатор — ОТДЕЛЬНЫЙ субагент с чистым контекстом, не тот же, что исполнял задачу. Вход строго ограничен:

- спека (GitHub issue — office-конвейер, или файл `.office/tasks/<slug>.md` — /solve);
- результат исполнителя (дифф, отчёт).

Он исполняет команду/сценарий, прописанный в DoD спеки, и возвращает **факт** — «команда X вернула Y» — а не оценку вроде «выглядит нормально» или «должно работать». Ничего не додумывает и не читает сверх спеки+результата.

## Лестница провалов

1. **Провал 1–2** — доработка тому же исполнителю с точечным списком расхождений (что именно не сошлось со спекой).
2. **Провал 3** — свежий исполнитель + диагноз верификатора (переданный как факт, не как «попробуй ещё»).
3. **Провал 4** — эскалация:
   - issue-носитель: лейбл `status:blocked`;
   - файловый носитель (`/solve`): поле `status: blocked` во frontmatter спеки;
   - в обоих случаях — пользователю короткий диагноз, конвейер едет дальше по независимым задачам (эта задача не блокирует остальные).

## Чек-лист выхода

- [ ] Вердикт содержит команду и её фактический вывод (не пересказ, не оценку).
- [ ] Провал → точечный список конкретных расхождений со спекой, а не общее «не работает».
- [ ] Спека требовала обновить доки → проверено отдельно, зафиксировано в вердикте.
- [ ] Носитель определён верно: issue → лейбл `status:*`; файл `.office/tasks/<slug>.md` → поле `status:` во frontmatter.
- [ ] На провале 4 диагноз пользователю умещается в несколько строк, конвейер не встал целиком.
